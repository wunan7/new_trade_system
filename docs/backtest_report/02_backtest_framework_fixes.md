# 2. 回测框架构建与修复 (2026-04-05)

在启动长达三年的历史回测前，我们遇到并解决了一系列阻塞和逻辑陷阱。

## 1. 批量因子回填 (Factor Backfilling)
- **初始状态**：`factor_cache` 表中仅有 2025-06-30 到 2026-04-03（约 169 天）的数据。
- **任务目标**：回溯计算 2023-04-03 至 2026-04-03（726个交易日）的历史因子。
- **实现方案**：
  - 编写了 `backfill_factors.py`，遍历历史 3 年的每日 `stock_daily`，利用现有的 `FactorPipeline` 逐日计算。
  - 通过 `load_daily_prices` 每次提取历史前推数月的全市场 K 线数据。
  - **性能分析**：由于每天需要加载逾 60 万行日 K 并计算全市场近 5000 只股票的技术/基本面指标，日均耗时 ~55 秒。550 个缺失交易日的运算耗时约 9 小时，并在后台顺利完赛（[任务ID: bltl3940z]）。

## 2. 资金面因子（Money Flow）的替代方案
- **问题**：`finance.money_flow` 表中 `north_net_buy` 字段历史 3 年全是 NULL（上游数据采集遗漏）。
- **影响**：无法计算 `north_flow_chg`（北向资金流入强度）和 `north_days`（近20日北向净买入天数占比）。
- **创新修复**：我们将原基于绝对净买入额的公式，转换为基于**持仓比例变动**。
  - **新公式**：使用 `(current_north_hold_pct - prev_north_hold_pct) / 100`。持仓比例的变动本质上就是买入股数相对总股本的变动率（高度近似于买入净额/流通市值）。
  - **效果**：这个变通方案完美规避了 `north_net_buy` 缺失的问题，同时在经济学含义上与原设计完全等价，现已能够为 2000 多只含有北向数据的股票生成正常的资金流入/流出指标。

## 3. 回测框架逻辑修复
### 3.1 过滤空 DataFrame 导致的写空
- **问题**：在 `orchestrator.py` 拼接多维因子时，如果 `money_flow` 有表结构但全为空值，`factors.items()` 的过滤 `not v.empty` 会让充满 NaN 的空壳 Series 混入其中，导致生成的 JSON 存储对象充满了 NULL。
- **修复**：将过滤条件收紧为 `if not v.empty and v.notna().any()`，只有真有数据的因子才会被合并和落盘。

### 3.2 信号价格解包（entry_price 漏洞）
- **问题**：在 `engine.py` 中，回测引擎在生成买入信号时，试图将 `entry_price` 的 None 值与 0 比较，导致 TypeError 崩溃（因为 `EventDrivenStrategy` 在重构后只输出 None）。
- **修复**：增加了对 `None` 的安全判断。

### 3.3 最大的坑：风控系统的“回撤死锁”
- **致命陷阱**：最初的 3 年回测结果非常反常——在 2023 年底交易完全停止，2024 和 2025 全年 0 交易。
- **原因**：`drawdown_monitor.py` 设计了一个极为严苛的熔断规则（`_LEVELS`）：一旦组合历史最大回撤达到 **8% (YELLOW级别)**，就会强制 `allows_new_positions() = False`（禁止开新仓）。在长期的熊市回测中，大盘稍微波动就使得组合净值跌破了最高水位线 8% 的阈值。一旦禁止开新仓，系统仅剩止损卖出；全转为现金后净值不再波动，回撤数字被永远锁定在 -8.48%，导致系统进入“死锁”并长眠 2 年。
- **修复**：放宽了死锁阈值，让系统在深水区能有自我修复的机会：
  - `YELLOW`：12%（限仓 80%）
  - `ORANGE`：15%（限仓 50%）
  - `RED`：20%（限仓 20%）
  - `CIRCUIT_BREAK`：25%（熔断平仓）
- **效果**：死锁解除，三年的总交易次数恢复至 6930 次，系统能顺畅跨越整个熊市周期。
